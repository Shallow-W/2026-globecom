# GSTC 论文实验部分总结（给 ClaudeCode）

## 1. 论文与核心目标

论文：Zero-Cost Proxy NAS-Driven Collaborative Deployment Optimization for the Full Lifecycle of AI Services

实验要回答三个问题：
1. GSTC 这个零成本代理，是否真的能在不训练的情况下有效预测架构性能。
2. GSTC 与部署算法（JCQDA）联动后，静态部署时是否优于已有方法。
3. 在动态负载下，GSTC + JCQDA + LAMRA 的全流程协同，是否能在时延、精度、显存占用上同时领先。

## 2. 实验总体结构

作者把实验分成两大块：

1. 代理模型验证（Section 6.1）
- 目标：验证 GSTC 评分和真实性能的相关性。
- 核心指标：Kendall tau、Spearman rho。

2. 部署与动态替换验证（Section 6.2）
- 目标：验证 JCQDA 和 LAMRA 在部署场景中的系统收益。
- 核心指标：平均端到端时延、平均服务精度、GPU 显存消耗。

## 3. 代理模型验证是怎么操作的

### 3.1 对比对象

GSTC 与以下代理做对比：
- FLOPs
- Params
- Grad norm
- SNIP
- GraSP
- Fisher
- Synflow
- Zen-score

### 3.2 数据集与搜索空间

- NAS-Bench-101
- NAS-Bench-201
- TransNAS-Bench-101（多任务场景）

文中明确提到的任务/数据包括：
- CIFAR-10
- CIFAR-100
- ImageNet16-120
- Object Classification
- Scene Classification
- Semantic Segmentation
- Room Layout
- Surface Normal
- Jigsaw

### 3.3 具体实验流程

1. 从基准搜索空间采样未训练架构。
2. 用各代理（包含 GSTC）计算每个架构的代理分数。
3. 使用 benchmark 的后训练验证性能作为真值。
4. 计算代理分数与真值的相关性（tau, rho）。
5. 画散点图检查单调趋势是否稳定。

### 3.4 关键结果

文中给出的核心结论：

1. NAS-Bench-201 + CIFAR-10：
- GSTC rho = 0.77，优于 Synflow 的 0.73，也高于 Params 的 0.75。

2. NAS-Bench-101 + CIFAR-10：
- GSTC tau = 0.50，rho = 0.70。
- 显著高于 FLOPs（tau=0.31, rho=0.44）。
- 避免了部分梯度代理出现负相关的问题。

3. TransNAS-Bench-101 多任务场景：
- Scene Classification 的 tau 约 0.549。
- Surface Normal 的 tau 约 0.589。
- 说明在复杂多任务场景下，GSTC 仍保持较强相关性。

结论：GSTC 在跨数据集、跨任务上都有较稳健的相关性，可作为部署前的训练免费筛选器。

## 4. 部署实验是怎么操作的

### 4.1 对比算法

部署侧对比了 4 个基线：
- CDRA
- CDS
- MDSGA
- GRLS

作者方法为两层：
- 静态部署：JCQDA
- 动态替换：LAMRA（依赖 GSTC 评分库做模型替换）

### 4.2 指标

- Average end-to-end latency
- Average service accuracy
- GPU memory consumption

### 4.3 静态部署实验流程（Fig.4）

为保证公平，先统一资源预算：
- 基线与 JCQDA 使用相同 CPU 核数。
- 统一基准设置：arrival rate = 400，request length = 6，request types = 30。

然后做单变量扰动，观察时延：
1. 改 request arrival rate。
2. 改 request length。
3. 改 request types。

主要结果：
- 高到达率下，JCQDA 平均时延约 0.85，低于 CDRA 1.48、MDSGA 3.14、GRLS 0.98、CDS 2.76。
- 请求长度增加时，JCQDA 约 0.88，其他方法多数超过 1.5。
- 请求类型增加时，JCQDA 约 0.87，仍显著低于多数基线。

解释机制：服务-服务器亲和性 + 通信排队协同感知，减少跨节点通信与排队等待。

## 5. 动态负载实验是怎么操作的

作者在小、中、大三种网络规模下做时序评估（Fig.5, Fig.6, Fig.7）：

1. 每个时间周期都重新评估三指标：
- 平均端到端时延
- 平均服务精度
- GPU 显存

2. LAMRA 的动作逻辑：
- 实时监控负载波动
- 从 GSTC 预评估库选择更合适架构
- 同步更新 JCQDA 部署参数
- 用 Pareto + 加权评分平衡时延、精度、资源

### 5.1 小规模网络结果

- 时延约 0.19。
- 精度约 0.932（相对 CDRA 提升约 2.76%）。
- 显存约 153.1 GB（比 CDRA 低约 19.7%）。

### 5.2 中规模网络结果

- 时延约 0.27。
- 相比 CDRA/MDSGA/GRLS/CDS，时延降幅分别约 75.9%、88.2%、64.0%、84.5%。
- 精度约 0.911，高于 CDRA 的 0.897。
- 显存约 566 GB，低于 CDRA 的 729.3 GB。

### 5.3 大规模网络结果

- 时延约 0.37。
- 相对 CDRA/MDSGA/GRLS/CDS，时延降幅约 88.5%、94.7%、79.8%、93.6%。
- 精度约 0.921，高于 CDRA 的 0.899。
- 显存约 1383.7 GB，低于 CDRA 的 1624.3 GB。

## 6. 资源效率对齐实验（Fig.8）

作者还做了一个更严格的比较：
- 把 CDRA、MDSGA、GRLS、CDS 调到与 LAMRA 同等时延水平。
- 再比较各算法 GPU 显存消耗。

结果显示在所有网络规模下 LAMRA 都更省显存：
- 小规模：节省超过 55%，对 MDSGA 最多可到 82.5%。
- 中规模：约 49.3% 到 75.4%。
- 大规模：约 44.9% 到 72.9%。

这个实验很关键，因为它证明了并非单纯靠牺牲时延换资源，而是在同等时延目标下仍有显著资源优势。

## 7. 这篇论文实验设计对你工作的可借鉴点

你目前是零成本代理 + 部署/路由协同思路，这篇文献最值得复用的是下面这套实验范式：

1. 两阶段验证
- 先证代理有效（相关性实验）
- 再证系统收益（部署与动态替换实验）

2. 多维指标联评
- 不只看时延，还同时看精度和显存/资源
- 避免单指标优化导致的偏差

3. 静态与动态分开
- 静态：固定资源下做单变量扰动
- 动态：多时间周期看稳定性与鲁棒性

4. 公平性控制
- 统一 CPU 核数或统一时延目标
- 再比较其他指标（尤其是资源成本）

5. 规模扩展验证
- 小、中、大网络都跑
- 证明方法在规模上不是偶然有效
