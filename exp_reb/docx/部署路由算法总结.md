**作者**: Yi Hu, Hao Wang, Liangyuan Wang, Menglan Hu, Kai Peng, Bharadwaj Veeravalli

### 1. 系统模型与问题建模

#### 1.1 网络拓扑与处理器模型

- **处理器集合** V = {v₁, v₂, ..., vₙ}，每个处理器 v 有 Cᵥ 个 CPU 核心
- 每个核心只能虚拟化一个容器，一个容器只能部署一个微服务实例
- 链路 evi,vⱼ = (Dvi,vⱼ) 属于集合 E，Dvi,vⱼ 表示两个处理器之间的传播延迟
- 当两个处理器相同时 Dvi,vⱼ = 0，不同时 Dvi,vⱼ > 0

#### 1.2 微服务模型

- 微服务集合 M = {m₁, m₂, ..., mₘ}
- 用户服务请求 r ∈ R 可建模为请求流 fr = (Mr, vfr, λfr)
  - Mr = {m₁, m₂, ..., m|Mr||mi ∈ M} 表示请求需要遍历的微服务节点集合
  - vfr 表示请求流的等效入口处理器
  - λfr 表示请求到达率，服从 Poisson 过程
- Nmᵥ ∈ ℝ 表示分配给处理器 v 上微服务 m 的核心数量
- xmᵥ ∈ {0,1} 表示微服务 m 是否部署在处理器 v 上

#### 1.3 M/M/C 排队模型

对于部署在处理器 v 上的微服务 m：

- **服务强度**: ρmᵥ = λmᵥ / (Nmᵥ · Λm)
  - λmᵥ 是微服务 m 在处理器 v 上的总到达率
  - Λm 是微服务 m 的单个核心处理速率（服从指数分布）
- **平均响应时间** (包含排队等待延迟和处理延迟):
  ```
  Wmv = ρmᵥ / (λmᵥ · (1 - ρmᵥ)² · pmᵥ) + 1 / Λm
  ```
  其中 pmᵥ 是等待概率

#### 1.4 请求路由模型

- P(r)mi,vi'^{mj,vj'} 是请求路由决策变量，表示请求从部署在处理器 vi' 上的微服务 mi 路由到部署在 vj' 上的微服务 mj 的概率
- 对于每个请求流，定义路由路径集合 RP(f) = {f₁, f₂, ..., f|RP(f)|}
- 每个路由路径 fₖ = {yfₖ^{m₁}, yfₖ^{m₂}, ..., yfₖ^{m|Mr|}} 记录请求经过的处理器序列

### 2. 部署算法详解 (GMDA)

**GMDA (Greedy-based Microservice Deployment Algorithm)**

#### 算法步骤

```
输入: 处理器计算资源集合 C; 每个微服务总计算资源需求 NM; 每个处理器剩余计算资源 ReC
输出: 每个微服务的二维放置结果 X = {xmᵥ, Nmᵥ}

1: 初始化: ReC = {C}, M_List = {M}, Free_List = {V}, Activate_List = φ, Available_V(m) = φ

2: 将 M_List 中所有微服务按 NM 降序排序

3: while M_List ≠ φ do
4:     获取 M_List 中第一个微服务 m，重置 Available_V(m) ← φ

5:     // 优先在使用中的处理器中查找
6:     搜索 Activate_List，将满足 ReCᵥ ≥ (1/ω)·Nm 的每个处理器 v 加入 Available_V(m)

7:     if Available_V(m) = φ then
8:         // 在空闲处理器中查找
9:         搜索 Free_List，将满足 ReCᵥ ≥ Nm 的每个处理器 v 加入 Available_V(m)
10:    end if

11:    if Available_V(m) = φ then
12:        // 放宽条件，查找部分资源
13:        搜索 Free_List，将满足 ReCᵥ ≥ (1/ω)·Nm 的每个处理器 v 加入 Available_V(m)
14:    end if

15:    if Available_V(m) = φ then
16:        微服务部署失败; continue
17:    end if

18:    // 按剩余资源量和通信延迟升序排列
19:    将 Available_V(m) 中所有处理器按 ReCᵥ 和通信延迟 Dvi,vj 升序排序

20:    for vi in Available_V(m) do
21:        if ReCvi - Nm ≥ 0 then
22:            // 资源足够，整数部署
23:            将微服务 m 部署在处理器 vi 上
24:            xm{vi} ← 1, Nm{vi} ← Nm, ReCvi ← ReCvi - Nm
25:            将微服务 m 从 M_List 移除
26:            若处理器 vi 来自 Free_List，则将其移动到 Activate_List
27:            break
28:        end if

29:        if ReCvi - Nm < 0 then
30:            // 资源不足，部分部署
31:            将微服务 m 部署在处理器 vi 上
32:            xm{vi} ← 1, Nm{vi} ← ReCvi
33:            若处理器 vi 来自 Free_List，则将其移动到 Activate_List
34:            Nm ← Nm - ReCvi  // 更新剩余需求
35:        end if
36:    end for

37:    从 M_List 移除微服务 m，更新 ReC, NM
38: end while

39: return X
```

#### 关键设计要点

1. **资源需求计算**:
   - 每个微服务 m 需要的资源量 Nm 由公式 Nm = Σ{f∈Mf} Nm(f) 计算
   - Nm(f) 是在最大延迟约束下微服务 m 处理请求流 f 所需的计算资源

2. **资源分割因子 ω**:
   - 允许微服务部署时进行资源分割
   - 当一个处理器无法提供全部 Nm 个核心时，可分割部署到多个处理器

3. **处理器选择优先级**:
   - 第一优先：Activate_List 中剩余资源 ≥ (1/ω)·Nm 的处理器
   - 第二优先：Free_List 中剩余资源 ≥ Nm 的处理器
   - 第三优先：Free_List 中剩余资源 ≥ (1/ω)·Nm 的处理器

4. **部署目标**:
   - 集中部署微服务到少数处理器
   - 优先利用已有处理器的剩余资源
   - 减少跨处理器通信延迟

### 3. 路由算法详解 (RMPR)

**RMPR (Request-Matching and Partition-Routing) Algorithm**

#### 算法步骤

```
输入: 放置结果 X = {xmᵥ, Nmᵥ}; 使用微服务 m 的所有请求流集合 fm;
      每个微服务的分区数量 multi_flow_List; 请求流到达率 λ

输出: 每个服务请求的路由结果 Yf^m; 转发概率集合 p(r)mi,vi'^{mj,vj'}

1: 初始化: recover_List = φ, Normalize_values = φ

2: // 聚类阶段：将请求流分组
3: while multi_flow_List 有超过一个分区 do
4:     将 multi_flow_List 中所有元素降序排序
5:     将 multi_flow_List[0] 和 multi_flow_List[1] 反向合并，得到新元组 temp_multi_flow_List
6:     将两个原始元组加入 recover_List，按所有值降序排序 temp_multi_flow_List
7:     归一化 temp_multi_flow_List，将减去的归一化值加入 Normalize_values
8:     用归一化的 temp_multi_flow_List 替换两个原始元组
9: end while

10: // 构建二叉树分区
11: 使用回溯方法，基于 multi_flow_List, recover_List 和 Normalize_values
    构建升序二叉树分区 Tuple(fm)

12: // 分配阶段：将请求流分配到处理器
13: 将 X(Nmᵥ) 降序排序，得到可用资源集合 {Nmᵥ}

14: while Tuple(fm) ≠ φ do
15:     获取 Tuple(fm)[0] 中的第一个请求流分区
16:     for Nm{vi} in {Nmᵥ} do
17:         if Nm{vi} ≥ ⌈λfm[i] / Λm⌉ then
18:             // 资源足够，完整分配
19:             记录分配给 fm[i] 在处理器 vi 上的资源为 ⌈λfm[i] / Λm⌉
20:             yfm[i]^m ← vi, Nm{vi} ← Nm{vi} - ⌈λfm[i] / Λm⌉
21:             移除请求流; break
22:         end if

23:         if Nm{vi} ≠ 0 && Nm{vi} < ⌈λfm[i] / Λm⌉ then
24:             // 资源不足，部分分配
25:             记录分配给 fm[i] 在处理器 vi 上的资源为 Nm{vi}
26:             yfm[i]^m ← vi, Nm{vi} ← 0
27:             λfm[i] = λfm[i] - Nm{vi} · Λm  // 更新剩余到达率
28:         end if
29:     end for
30: end while

31: // 计算转发概率
32: 根据 Yf^m 和 Nmᵥ 公平分配所有请求权重比例
33: return Yf^m 和 p(r)mi,vi'^{mj,vj'}
```

#### 关键设计要点

1. **聚类分区方法**:
   - 假设微服务 m 部署在三个处理器上，则每个请求流被分成三个分区
   - 初始状态：每个请求流作为独立元组，剩余值为0
   - 合并规则：给定两个元组 (x₁,y₁,z₁) 和 (x₂,y₂,z₂)，合并为 (x₁+z₂, y₁+y₂, z₁+x₂)

2. **回溯构建二叉树**:
   - 利用 recover_List 和 Normalize_values 恢复原始分区
   - 确保请求流被公平且有效地分配到各处理器

3. **转发概率计算**:
   - YF^M 是一个 |F| × |M| 矩阵，记录每个请求流遍历每个微服务时的路由目标
   - 结合 Nmᵥ 确定最终转发概率 P(r)mi,vi'^{mj,vj'}

### 4. 实验设置与Baseline策略

#### 4.1 实验设置

**微服务与调用图**:
- 微服务数量规模：2 到 50 个
- 请求数量范围：10 到 1000
- 每条请求遍历的调用链最多包含 14 个微服务，链长范围 2 到 9

**网络拓扑**:
- 数据中心网络拓扑（fat-tree 等）
- 处理器数量：10 到 64 个
- 每个处理器资源容量（核心数）：8 到 64
- 处理器间通信延迟不超过用户容忍延迟的 5%

**请求流参数**:
- 到达率服从 Poisson 分布，范围 1 到 10
- 单个核心处理速率 Λm 对每个微服务在 1 到 5 之间变化

#### 4.2 Baseline算法

1. **RLS (Random Local Search)**:
   - 采用贪心部署策略
   - 在公平概率路由规则的邻域内搜索路由策略
   - 迭代次数最少（平均 12.9 次）

2. **FFD (Modified First Fit Decreasing)**:
   - 传统的首次适配递减算法
   - 按资源需求降序排列微服务，依次部署到第一个满足条件的处理器
   - 迭代次数最少（平均 9.7 次）

3. **DRS (Dynamic Resource Scheduling)**:
   - 具有概率请求路由的动态资源调度算法
   - 需要在大可行空间中搜索
   - 迭代次数最高（平均 359.1 次）

4. **LEGO**:
   - 三阶段算法
   - 联合优化服务请求路由和实例放置
   - 迭代次数平均 17.5 次

#### 4.3 性能指标与结果

| 指标 | GMDA-RMPR 表现 |
|------|----------------|
| 平均部署成本降低 | 27.4%（相比所有baseline） |
| 端到端响应延迟降低 | 15.1%（平均） |
| 请求成功率 | 当资源充足时比baseline提高 5.47% |
| 使用处理器数量 | 相比 DRS 减少 63.2%，相比 RLS 减少 32.7% |

**详细性能对比**:
- RLS: 延迟改善 7.26% ~ 26.7%
- FFD: 平均延迟改善 19.2%
- DRS: 延迟改善约 4%
- LEGO: 延迟改善约 8.1%

## 论文二：Joint Task Offloading, Resource Allocation and Model Placement for AI as a Service in 6G Network

**来源**: IEEE TRANSACTIONS ON SERVICES COMPUTING, VOL. 17, NO. 6, NOVEMBER/DECEMBER 2024

**作者**: Yuhao Chai, Kaice Gao, Guohan Zhang, Lu Lu, Qin Li, Yong Zhang

### 1. 系统架构

#### 1.1 框架组成

- **边缘-网络-云框架**: 配置 AIaaS (AI as a Service) 功能单元
- **可配置功能单元**:
  - Service Split Manager: 将有依赖的任务划分为单目的地节点任务序列
  - Service Quality Manager: 管理服务 QoS 需求（CPU/GPU 核心数、最大延迟、精度要求）
  - Model Complexity Manager: 管理模型仓库和 AI 模型参数
  - Decision Manager: 定期更新模型放置策略，设计任务卸载和资源分配策略

#### 1.2 决策变量

- βi,j,k(t) = 1: 任务 i 在节点 j 上使用模型 k 进行计算
- WC{i,j}(t): 分配给 CPU 的任务工作量
- WG{i,j}(t): 分配给 GPU 的任务工作量
- fC{i,j}(t): CPU 最优工作频率
- fG{i,j}(t): GPU 最优工作频率
- αj,k(t) = 1: 模型 k 部署在节点 j 上

---

### 2. 任务卸载算法详解 (DA)

**DA (Deferred Acceptance) Algorithm - 基于 Gale-Shapley 算法**

#### 权重计算

任务 i 与节点 (j,k) 之间的权重:
```
wi,j,k = V2·(μ/T*)·(di,j·Ci/Ri,j + DCom*i)
        + V1·Zi,j·(bi,k·L·Ci - αj,k·Tslot·[γC_i·fC{i,j} + γG_i·fG{i,j}])
        + V2·(1-μ)/E*·(εd_i·di,j·Ci/Ri,j + εC_j·WC{i,j}/fC{i,j}² + εG_j·WG{i,j}/fG{i,j}²)
```

#### 算法步骤

```
1: Application (申请阶段):
   - 每个任务根据偏好列表 ψi = {wi,j,k(t)} 选择最优计算节点
   - 每个任务只能向从未拒绝过它的计算节点申请

2: Selection (选择阶段):
   - 每个计算节点 j 收到申请者集合 φj
   - 由于同一节点的模型共享计算容量，合并所有模型 k 的偏好列表 ψj,k 为节点偏好列表 ψj
   - 根据 ψj，每个节点按升序接受一定数量的任务
   - 拒绝的任务权重设为 wmax (惩罚阈值)

3: Deferred Acceptance (延迟接受):
   - 被拒绝的任务继续向下一优选节点申请
   - 直到所有任务被接受或无法再申请
```

#### 贪心路由策略

论文中提到的"贪心分配异构资源，将服务实例部署在资源利用率最低的节点；路由时始终选择当前位置最近的节点转发请求"具体指:

- **部署**: 将服务实例部署在资源利用率（CPU/GPU）最低的节点，追求局部最优
- **路由**: 始终选择当前地理位置最近的节点转发请求，最小化通信开销

---

### 3. 资源分配算法

#### 3.1 卸载权重最小化

使用 Lyapunov 优化将长期问题转化为短期优化问题，目标函数:
```
min F2 = Σi Σj (Xi,j + Yi,j + Gi,j + Hi,j)
```

#### 3.2 最优工作负载分配

根据 Lemma 1 和 Lemma 2，最优工作负载分配为:
```
WC*{i,j}(t) = Σk β{i,j,k}(t)·bi,k·L·Ci / (√(εG_j·γC_i) / √(εC_j·γG_i) + √(εG_j·γC_i))
WG*{i,j}(t) = Σk β{i,j,k}(t)·bi,k·L·Ci / (√(εC_j·γG_i) / √(εC_j·γG_i) + √(εG_j·γC_i))
```

### 4. 模型放置算法 (MAB-UCB)

#### 算法步骤

```
输入: 节点 j 的模型选择臂集合 Kj
输出: 下一时间尺度的模型放置策略

1: // 初始化 UCB
2: 假设在每个时间槽 t ∈ [TT0, (T+1)T0) 内，进行 V 次尝试
3: 每次尝试随机改变节点模型组合

4: // 计算累计遗憾
5: 对于臂 (j,k):
   R̄j,k(t) = Σ(τ=TT0 to (T+1)T0-1) w̄(τ)vj,k(τ) - w*(t)
   其中 w̄(τ) 是选择臂 (j,k) 的平均权重，hj,k(t) 是选择次数

6: // UCB 公式
7: arm(j) = argmin_k [ R̄j,k(t) - √(2·log(V*(t-TT0)) / (hj,k(t)+1)) ]

8: // 选择最低遗憾的 αmax_j 个臂
9: 在 (t+1)T0 时刻，为每个节点选择遗憾最小的 αmax_j 个臂作为下一阶段模型放置策略
```

## 论文三：GSTC-JCQDA-LAMRA 框架

**来源**: GSTC 论文 - Kai Peng, Yue Yang, Jing Lu, Xinyi Yang, Yi Hu, Menglan Hu

---

### 1. 系统模型

#### 1.1 AI服务部署模型

- AI服务集合 M = {m₁, m₂, ..., mₘ}
- 服务版本集合 Km = {k₁, k₂, ..., k|Km|}
- 每个版本 k 有: 精度 ak、GPU内存消耗 CGPU_m,k、处理密度 Lk
- 服务器集合 V = {v₁, v₂, ..., vₙ}，每台服务器有 CPU核心 CCPU 和 GPU内存 CGPU

#### 1.2 M/M/C/K 排队模型

对于部署在服务器 v 上的服务 m:

- **服务强度**: ρv_m = λv_m / (Nv_m · μm)
- **稳态概率**: 使用 Birth-Death 过程计算
- **平均队列长度**: Lq = (ρ²·P₀) / (c·(1-ρ))

#### 1.3 端到端延迟模型

对于请求 r:
```
Tr = Σi∈Mr ts_{mi} + Σi∈Mr-1 ttran_{mi,mi+1}
   = Σi∈Mr (1/μm_i) + Σi∈Mr-1 (data(mi,mi+1) / Bva,vb)
```

其中:
- ts_{mi}: 服务 m_i 的处理延迟
- ttran_{mi,mi+1}: 服务间传输延迟
- data(mi,mi+1): 服务间传输数据量
- Bva,vb: 服务器间链路带宽

---

### 2. GSTC 零成本代理模型

#### 2.1 信息量量化编码

GSTC 使用四个维度评估神经网络架构:

1. **G (总路径信息)**: 架构的总信息量
2. **S (路径变化)**: 路径信息差异量 |g₁-g₂|, ..., |g₃-g₄|
3. **T (输出节点接收的总信息)**: 输出节点接收的信息量
4. **C (输出变化)**: 输出信息差异量 |t₁-t₂|, ..., |t₃-t₄|

#### 2.2 代理分数计算

```
Score = a·G + b·S + p·T + q·C
```

通过差分训练基于排名相关性学习参数 [x, y, z, a, b, p, q]

#### 2.3 与其他代理模型对比

### 3. JCQDA 部署算法

**JCQDA (Joint部署算法)**

#### 算法步骤

```
1: // 计算最小实例数（保证系统稳定性）
2: Nm = ⌈(1+θ)·λm / μm⌉
   其中 θ ≥ 0 是松弛因子，吸收突发流量

3: // 计算服务-服务器亲和度
4: Avm = α·Availvm + β·PSvm
   - Availvm: 服务器 v 上可部署的逻辑服务 m 的实例数
   - PSvm: v 上依赖服务 m 的现有实例数量
   - α, β: 超参数，平衡排队延迟减少和通信延迟减少的收益

5: // AI服务部署
6: 将服务器按 Avm 降序排列
7: 迭代放置实例：直到 Nm 个实例放置完成
8: 每次迭代在最高亲和度服务器上放置一个实例
```

#### 关键设计要点

1. **服务-服务器亲和度**:
   - 同一服务多实例部署在同一服务器可降低排队延迟
   - 顺序依赖服务共置可减少跨服务器传输
   - 两者存在冲突，亲和度指标平衡利弊

2. **部署约束**:
   - CPU约束: Σm Nm·CCPU_m ≤ CCPU
   - GPU约束: Σm Nm·CGPU_m ≤ CGPU

### 4. LAMRA 动态模型替换算法

**LAMRA (Load-Aware Model Replacement Algorithm)**

#### 算法步骤

```
输入: 到达率变化后的请求集合 Rchange
输出: 新的微服务版本集合 Mchange

1: Mchange = {}
2: for m ∈ M do
3:     AvailKm = {}
4:     // 计算新的请求到达率总和
5:     计算微服务 m 的新请求到达率总和 λm

6:     // 当服务强度小于1时，计算满足QoS的最小处理速率
7:     当 λv_m / (Nm·μm) < 1 时:
8:         计算 μm_min = λm / Nm

9:     for k ∈ Km do
10:        if μm,k ≥ μm_min then
11:            AvailKm = AvailKm ∪ {mk}
12:        end if
13:    end for

14:    // Pareto前沿筛选
15:    对 AvailKm 中的版本执行 Pareto 筛选，得到前沿版本集合 FrontierKm

16:    for mk ∈ FrontierKm do
17:        // 计算综合评分
18:        计算版本 k 的评分
19:    end for

20:    // 选择最优版本替换
21:    选择评分最高的版本 mbest 替换原版本
22:    Mchange = Mchange ∪ {mbest}
23: end for

24: return Mchange
```

#### 触发条件

- 当请求强度变化超过预设阈值时触发

#### Pareto前沿筛选

- 考虑精度和GPU内存消耗两个目标
- 选择帕累托最优版本集合

#### 5.2 Baseline算法

1. **CDRA (请求优先级部署)**:
   - 按请求优先级部署服务
   - 通过增加实例数量满足延迟约束
   - 缺乏架构级优化

2. **CDS (协同微服务部署)**:
   - 尝试将同一请求链的微服务共置在单服务器
   - 减少跨服务器通信
   - 但可能造成负载不均衡

3. **MDSGA (改进编码遗传算法)**:
   - 通过改进编码和灾变机制避免局部最优
   - 随机初始化实例数量
   - 迭代调整

4. **GRLS (贪婪本地搜索)**:
   - 贪婪构建初始解
   - 通过本地搜索细化
   - 根据请求量和处理率设置实例数量

## 三篇论文算法对比总结

| 维度 | 论文一 | 论文二 | 论文三 |
|------|--------|--------|--------|
| **部署策略** | GMDA: 按资源需求降序，部署到通信距离最短的处理器 | 贪心分配到资源利用率最低节点 | JCQDA: 基于M/M/C排队模型+服务-服务器亲和度 |
| **路由策略** | RMPR: 聚类分区映射+概率路由 | 选择最近节点转发（贪心） | 基于排队模型优化路由 |
| **动态调整** | 静态两阶段优化 | MAB-UCB模型放置更新 | LAMRA负载感知模型替换 |
| **核心排队模型** | M/M/C | 考虑传输+计算+排队延迟 | M/M/C/K |
| **优化目标** | 部署成本+响应延迟+成功率 | 延迟+能耗加权和 | 资源利用率+延迟+QoS |
| **特色机制** | 微服务复用+容器共享 | 双时间尺度Lyapunov优化 | NAS零成本代理GSTC |

### 论文一: 部署核心流程

1. 按资源需求降序排列微服务
2. 优先使用已有处理器剩余资源
3. 选择通信距离短的处理器
4. 允许资源分割部署

### 论文二: 部署核心流程

1. 任务-节点匹配基于权重wi,j,k
2. 贪心选择资源利用率最低节点部署
3. 路由始终选最近节点
4. MAB-UCB动态更新模型放置

### 论文三: 部署核心流程

1. 计算最小实例数保证稳定性
2. 评估服务-服务器亲和度
3. 按亲和度降序迭代放置
4. LAMRA根据负载动态替换模型版本
