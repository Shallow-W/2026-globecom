6.2 Deployment Model Validation Results We evaluate the proposed JCQDA deployment algorithm and the LAMRA dynamic replacement algorithm against four baselines: CDRA, CDS, MDSGA, and GRLS, using average end-to-end latency, service accuracy, and GPU memory consumption as metrics. Among these, CDRA [44] deploys services by request priority, satisfying latency constraints by increasing instance counts. CDS [45] attempts to co-locate microservices from the same request chain on a single server to reduce cross-server communication. MDSGA [46] avoids local optima via improved encoding and a catastrophe mechanism, randomly initializing instance counts and iteratively adjusting them. GRLS [47] constructs an initial solution greedily and refines it by local search, setting instance counts based on request volume and processing rate. To ensure fair comparison in static deployment, all baselines use the same number of CPU cores as JCQDA, with a unified setting of arrival rate 400, request length 6, and 30 request types, and Figs. 4a–4c illustrate how latency varies with these variables. Under higher arrival rate, JCQDA achieves 0.85 average latency, versus 1.48 (CDRA), 3.14 (MDSGA), 0.98 (GRLS), and 2.76 (CDS). When the request length increases, JCQDA remains at 0.88 latency, while most baselines exceed 1.5. When the number of request types grows, JCQDA still maintains 0.87 latency, while CDRA, MDSGA, and CDS increase to 1.49, 3.23, and 2.70, respectively. This comes from the “service-server affinity” design and the ‘communicationqueuing collaborative perception” mechanism, which reduce both cross-server transmission and waiting time. As a result, it maintains stable end-to-end latency, achieving efficient resource scheduling in static deployments. In addition, we conduct dynamic scenario validation in small, medium, and large-scale networks respectively.

6.2.1 Small-Scale Networks Under dynamic load in small-scale networks, LAMRA outperforms CDRA, MDSGA, GRLS, and CDS on average end-to-end latency, service accuracy, and GPU memory consumption. In Fig. 5a, it achieves 0.19 average latency, 38% lower than all baselines, 62.7% lower than MDSGA and 38.7% lower than GRLS. In Fig. 5b, LAMRA attains 0.932 average accuracy (2.76% higher than CDRA). In Fig.5c, LAMRA achieves 153.1 GB average GPU memory consumption (19.7% lower than CDRA). The gains are attributed to the end-to-end pipeline combining GSTC zero-cost evaluation and JCQDA static deployment, which covers “load awareness, model replacement, and deployment linkage.’ GSTC rapidly selects “highaccuracy, low-resource” architectures without full training. LAMRA monitors load fluctuations in real time and chooses architectures guided by GSTC to maintain accuracy while controlling GPU memory consumption. During model replacement, JCQDA deployment parameters are simultaneously updated to avoid latency issues caused by improper deployment. At the same time, it employs Pareto frontier selection to balance multiple metrics. Conversely, baselines mainly adjust instance counts or container placement and lack architecture-level optimization.

6.2.2 Medium-Scale Networks In medium-scale networks, LAMRA remains ahead across the same three metrics: average end-to-end latency is 0.27 in Fig. 6a, which is 75.9%, 88.2%, 64.0%, and 84.5% lower than CDRA (1.12), MDSGA (2.28), GRLS (0.75), and CDS (1.74), respectively. In Fig. 6b, its average accuracy is 0.911, 1.56% higher than CDRA (0.897). In Fig. 6c, its GPU memory is 566 GB, 22.4% lower than CDRA (729.3). The dynamic adjustment mechanism of LAMRA makes its advantages particularly evident in medium-scale networks. On one hand, with servers and complex dependencies, JCQDA’s service–server affinity prioritizes placing dependent services on nodes with better links, sharply reducing cross-server latency. In contrast, the baselines lack such coordination, leading to a marked latency increase as the number of nodes grows. On the other hand, load fluctuations occur more frequently in medium-scale networks. With the evaluation system built on the informationquantity quantization encoding mechanism, GSTC can precisely match architectures to current load, avoiding accuracy loss or resource waste from poor architectural adaptation.

6.2.4 Resource Efficiency Across Networks To assess the resource efficiency of LAMRA, we fix CDRA, MDSGA, GRLS, and CDS to the same end-to-end latency as LAMRA, comparing GPU memory usage across network scales as shown in Fig. 8. The experiment shows that LAMRA delivers consistent GPU savings at all scales, over 55% in small-scale networks (even reaching 82.5% versus MDSGA), 49.3%–75.4% in medium-scale networks, and 44.9%–72.9% in large-scale networks. The GPU memory efficiency of LAMRA arises from the collaborative optimization of architecture screening, model selection, and deployment constraints via GSTC and JCQDA. GSTC embeds resource awareness at the design stage, identifying “performance-resource balanced” architectures, eliminating GPU redundancy at the source. LAMRA then employs Pareto frontier selection and weighted scoring to favor models with lower GPU memory under latency constraints. JCQDA deployment rules precisely assess server GPU capacity, preventing waste from overflow. In contrast, baselines mainly adjust instance counts or placement and therefore remain GPU-inefficient even when meeting latency requirements. End-to-end collaboration allows LAMRA to keep response speed while using GPU memory efficiently, further verifying the practicality of our fulllifecycle collaborative optimization across network scales.




A. Experiment Settings Microservices and call graphs: Luo et al. [6] summarized a variety of microservice call graphs and modeled them with tree structures, the vast majority of which are pipelined chains. For simplicity, we extract and preprocess independent pipelined request chains from the call graphs. Traced by this article, the number of microservices is typically scaled from 2 to 50. The number of requests ranges from 10 to 1000, and each request traverses a long call chain which consists of at most 14 microservices in our setup, where the length of chain mostly ranges from 2 to 9. Network topology and processors: Data center network topology (e.g., fat tree) has been investigated in [34], [35]. In our experiments, a full connection service mesh or connection graph is considered to model a data center, which contains from 10 to 64 processors. Besides, there is also sufficient switch capacity and network bandwidth to serve each request. The resource capacity (the number of cores) per processor is scaled from 8 to 64. The processing core of processors indicates the ability of handling a unit microservice requests, which serve different types of microservices at different speeds. Furthermore, as microservice-based requests are typically latency-sensitive, we believe that the maximum communication latency between two processors should not exceed 5 percent of the user-tolerated latency in a data center.


Distribution ofrequestflow rate andcontainers: In data center networks, [36], [37] show that the arrival rate of requests follows an interval time distribution. In our experiment configurations, it is assumed that the arrival rate ofrequest flows follows a Poisson distribution scaling from 1 to 10. It is worth noting that the same user request flow can arrive multiple times. Following the open Jackson queuing theory, each service instance of the same microservice has the same exponential service rate. Multiple service requests can be routed to the same containers, and a request can be routed to different processors and eventually merged. The service rate per core or container is less than the average arrival rate since microservice architectures typically require the deployment of hundreds of containers. To ensure that the simulation is scalable and large-scale, it is assumed that processing rate Λm of a single core varies from 1 to 5 for each microservice. 

Baseline algorithms: We choose two robust benchmark algorithms consisting of Random Local Search (RLS) [38] and modified First FitDecreasing (FFD) [39], [40], which arewidely used in resource management and combined with the GMDA in the experiment.We also implement a variantDRSalgorithm [41] with probabilistic request routing, the modification of which is to accommodate our joint optimization problem. Moreover, a three-stage algorithm LEGO [29] is used as a baseline for the joint deployment and routing, which adopts the deterministic routing policy and balanced deployment. To comprehensively evaluate the performance ofGMDA-RMPR, we compare it with them and calculate average values as simulation results through a large number of repeated experiments.
